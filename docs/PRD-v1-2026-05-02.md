## 1. Mission / 范围 / 非目标

### 1.1 一句话定义

> **ScoutFlow = 本地优先的多源内容采集 / 转写 / 规整 / 检索 / 主动推荐 / 知识关联 / 多 agent 协作中台。**

### 1.2 不是什么（非目标，明确剔除）

ScoutFlow **不是**以下任何一类，就算它们之间有重叠：

| 它不是 | 区别在哪 |
|---|---|
| 视频下载器（MeTube / yt-dlp UI） | 下载是**模块之一**，不是核心。ScoutFlow 不为"看视频"服务 |
| 自托管视频服务器（TubeArchivist / MyTube） | 不为"播放消费"服务。媒体文件是 evidence，不是 product |
| RSS 阅读器（FreshRSS / Miniflux） | RSS 是 source 之一，不是产品形态 |
| 知识库（Khoj / AnythingLLM / Open WebUI） | 知识关联是输出之一，不是产品形态。RAW / Obsidian 仍是知识权威 |
| 工作流引擎（n8n / Windmill / Kestra） | 不做任意自动化。领域模型固定（source / capture / transcript / link） |
| 小红书 / B 站自动化营销工具 | 严格只读、低频、个人研究用途。不评论、不点赞、不发布、不批量骚扰 |
| AI Chat 平台（Open WebUI 类） | 不以对话为核心交互形态 |

### 1.3 是什么（核心目标，按业务价值排序）

ScoutFlow **必须**同时做到以下 6 件事，才算 v1 成功：

1. **多源采集**：Bilibili + 小红书 一期正式源；YouTube 后置（Phase 2）；统一 capture record 抽象
2. **本地文案库**：每条采集落地为可检索 / 可引用 / 可时间戳定位的素材
3. **主动推荐**：4 类入口（账号 / 关键词 / 链接 / 列表）+ 规则召回 + LLM 重排
4. **知识关联**：与 RAW / Obsidian 双向关联（content → RAW patch / RAW gap → content）
5. **多 agent 协作**：Codex / Opus / Hermes / OpenClaw / ChatGPT Pro 通过 file-backed candidate 协作，user 拍板
6. **本地优先 + 离线可用**：核心闭环（采集 → 转写 → 入库 → 检索）不依赖外部 SaaS

### 1.4 不做（红线，违反即 reject）

- ❌ 绕过付费 / 权限 / 地区限制
- ❌ 代理池 / 反封锁 / 反检测增强
- ❌ 高频批量请求（任何源默认低频）
- ❌ 小红书自动评论 / 点赞 / 关注 / 发布
- ❌ 无授权再分发下载内容
- ❌ Hermes / OpenClaw 直接绕过 UI 频率限制
- ❌ 任何 agent 跳过 candidate 流程直接改主线 spec
- ❌ 把 ScoutFlow 部署成多用户 SaaS 服务

### 1.5 范围边界（Phase 1 内 vs Phase 1 外）

| 维度 | Phase 1 内 | Phase 1 外（后置） |
|---|---|---|
| 平台 | Bilibili / 小红书 | YouTube（Phase 2）/ Twitch / Podcast / 国内长视频站 |
| 入口 | 4 类全上（账号 / 关键词 / 链接 / 列表）+ 能力分级 | 浏览器扩展 / 移动端 / Apple Shortcuts |
| 采集 | metadata + media + audio + transcript | 字幕 OCR / 弹幕语义抽取 / 多机位合并 |
| 转写 | faster-whisper（默认）/ WhisperX（高级） | 实时流式转写 / 翻译 |
| 规整 | LLM 摘要 + 论点 + 引用 + 标签 | 自动选题生成 / 自动台本骨架（Phase 3+） |
| 推荐 | 规则召回 + LLM 重排 | 训练专属推荐模型 / 用户行为分析 |
| 知识关联 | RAW / Obsidian 单向 patch 建议 | 自动合并 RAW（永远人工 gate） |
| 多 agent | Codex / Opus + Hermes / OpenClaw 定时扫描 | 自治 agent 闭环（Phase 4） |
| 部署 | 单机本地 | 多机 / 远程 / 云端 |

### 1.6 与 DiloFlow 的边界（重要，避免治理污染）

ScoutFlow 与 DiloFlow 是**独立兄弟项目**。物理上：

- ScoutFlow 文件全部在 `/Users/wanglei/workspace/ScoutFlow/`
- DiloFlow 文件全部在 `/Users/wanglei/workspace/DiloFlow/`
- 不互相 cd / 跨项目 import / 跨项目 git submodule

逻辑上：

- ScoutFlow 不读 DiloFlow 的任何文件作为权威
- DiloFlow 可以**只读**消费 ScoutFlow 的 export 产物（normalized markdown / capture metadata），但要把它当作 external reference 处理，引用副本而非原件
- 双方各自维护 `decisions.md` / `current.md` / `tasks.md`，互不污染
- ScoutFlow 的 agent 协作规则与 DiloFlow 的可以**继承**，但每条规则在 ScoutFlow 侧必须重写一遍 lock 进 ScoutFlow 自己的 decisions.md（不允许跨项目 D-lock 引用）

## 2. 用户与场景

### 2.1 用户画像

ScoutFlow v1 的目标用户是**唯一一个**：项目所有者本人。

具体画像：

- 单人创作者 + 频道主理人（DiloFlow / 看见看不见频道）
- 多 agent 操作员（Codex / Opus / Hermes / OpenClaw / ChatGPT Pro 同时使用）
- 工作站：Mac M5
- 已有工具栈：BBDown（本机已通）/ yt-dlp（PATH 内）/ xhs-research skill / RAW 知识库 / Obsidian
- token 预算：Anthropic Max20 + ChatGPT Pro 200美元 + 火山 coding plan + GLM 5.1 / Codex CLI 等多模型路由
- 痛点：信息源散乱（YouTube + B 站 + 小红书 + 知乎 + 微信公众号 + RSS + 收藏夹）、采集到的内容沉不下来、选题灵感找不到证据支撑、多 agent 产出没有合并机制
- 工程偏好：file-backed authority、authority-first 架构、状态机思维、不接受隐性升级
- 不是"用户群"，**是"操作员 + 决策者 + Showrunner"三合一**

ScoutFlow v1 **不为多用户设计**。任何"协作 / 分享 / 多账号"功能都是 Phase 5+。

### 2.2 五条典型用户路径

#### P1 — 白名单订阅触发主动推荐

```text
user 在 Sources 页加 B 站 UP 主"影视飓风"为账号源（state=ready）
    ↓
Hermes 定时扫描（默认 6 小时）
    ↓
发现 UP 主新作品"AI 工具实测"
    ↓
Recommender 规则召回：白名单源 +时长 14min + 中文 → score 0.78
    ↓
LLM Reranker 用 RAW context（"AI Agent" / "内容生产" 节点已有 12 条相关内容）
    ↓
推荐分调到 0.86，理由："与 RAW『AI Agent 选型』节点高度相关，可能补充 PromptOpt 案例"
    ↓
推送到 Mission Control 的 Decision Dock 和 Source Radar
    ↓
user 在 Source Radar 卡片上点【采集 audio + transcript】
    ↓
触发 capture pipeline：metadata → audio download → ASR → normalize → index → RAW link 建议
    ↓
12 分钟后 Mission Control 提示"capture_id=01HXXX 已完成，3 个 RAW patch 候选"
    ↓
user 在 Knowledge Bridge 页审 patch，2 条 accept、1 条 reject
    ↓
accept 的 patch 写入 RAW inbox，等下一轮 RAW review 合并
```

**关键**：从订阅到 RAW 入库总耗时 < 20 分钟，user 实际操作 < 3 分钟（其余 worker 自动）。

#### P2 — 单条 URL 即时探针

```text
user 在微信看到一条 B 站视频链接，复制
    ↓
打开 ScoutFlow Explore 页粘贴 URL（也可走剪贴板捕获器自动识别）
    ↓
Explore 显示预览：标题 / UP / 时长 / 已知字幕 / 预估 ASR 时间 / 估算下载大小
    ↓
RAW 匹配预估：找到 2 个相似主题节点（"内容生产工具链"、"AI 视频工作流"）
    ↓
user 选择采集模式：[只 metadata] [audio + transcript] [video 全量]
    ↓
点【采集】→ 进入 Capture Pipeline 队列
```

**关键**：从粘贴到决定采集 < 30 秒，user 看到的是 metadata + RAW 匹配两件事，不是下载进度条。

#### P3 — 关键词扫描发现新趋势

```text
user 在 Sources 页加关键词源："AI Agent 编排" + "多模态推理" + "OpenAI o3" + 平台限定 [B 站, 小红书]
    ↓
Hermes 每 12 小时扫描一次（关键词扫描频率默认低于账号源）
    ↓
某次扫描命中 8 条新内容
    ↓
Recommender 规则召回：关键词命中 + 发布时间 < 7 天 + 互动数 > 阈值 → 5 条入候选
    ↓
LLM Reranker 重排，过滤掉 2 条标题党
    ↓
3 条进入 Source Radar，每条标推荐分 + 理由 + RAW 关联
    ↓
user 在 Source Radar 选择采集 / 加入观察 / 忽略
```

**关键**：关键词源是**发现工具**，不是采集源。命中物先进 Source Radar 等 user 决定，不直接采集（避免误采集 + 频率失控）。

#### P4 — 列表源批量入队

```text
user 在 Sources 页加列表源：B 站个人收藏夹"待研究"（已含 23 条视频）
    ↓
ScoutFlow 一次性导入 metadata 23 条 → 进入 Source Radar
    ↓
Recommender 给每条打分 + 排序
    ↓
user 在 Source Radar 批量选 12 条点【批量采集 audio】
    ↓
Capture Pipeline 队列接收 12 个 capture job，按并发限制（默认 2）逐批跑
    ↓
预计总耗时 1.5 小时（12 条 × 平均 8 分钟 / 并发 2）
    ↓
完成后 Copy Library 多 12 条文案，Mission Control 显示批次摘要
```

**关键**：列表源是**批量入口**。批量采集时必须显式确认（不允许"加列表源就自动采集全部"），否则一个 1000 条收藏夹就把磁盘塞满。

#### P5 — RAW 知识缺口反向触发

```text
user 在 RAW 写"AI 内容生产工作流"节点，发现"用户洞察阶段"段落没有外部证据
    ↓
ScoutFlow 在 Knowledge Bridge 页有 "Reverse Discovery" 入口
    ↓
user 输入：节点 path / 缺口段落 / 想找的证据类型（案例 / 数据 / 反例 / 框架）
    ↓
ScoutFlow 把缺口转成关键词 + 平台 + 时长偏好的 source candidate
    ↓
立即触发一次性扫描（不长期订阅）
    ↓
返回 6 条 Source Radar 候选
    ↓
user 选 2 条采集 + 转写 + 关联回 RAW 节点
```

**关键**：这是 ScoutFlow 区别于普通采集器的关键能力——**知识系统驱动采集**，不是采集驱动堆积。

### 2.3 反路径（明确不走的场景）

以下场景 ScoutFlow v1 **不支持**，需求来了 reject 不商量：

- ❌ 把 ScoutFlow 当作"看视频代理"（user 想看视频应该用原平台 app）
- ❌ 把 ScoutFlow 当作"批量下载到外部硬盘归档"（用 yt-dlp 直接跑，不要 ScoutFlow）
- ❌ 把 ScoutFlow 接入第三方平台分发（生成的 normalized 文档不允许公开分发）
- ❌ 让 ScoutFlow 自动改 RAW（RAW 写权永远在 user）
- ❌ 让 ScoutFlow 跨账号采集（一个 ScoutFlow 实例只服务一个 user 一组凭据）

## 3. 三份调研的统一吸收

ScoutFlow v1 PRD 的设计吸收了三份独立调研，本节交代每份的贡献和最终裁决。

### 3.1 调研① — 产品骨架（Codex 主笔）

**贡献**：把"采集线"从模糊概念升级为可命名的产品形态：

- **Mission Control** = 总控台（首页）
- **Module Forge** = 后端可插拔工坊
- **Evidence Ledger** = 数据库作为采集证据账本

并给出 9 页面 IA 草案（Mission Control / Source Radar / Explore / Capture Pipeline / Transcript Studio / Copy Library / RAW Link / Sources / Agents），每页核心组件清单。

**采纳**：9 页面 IA 整体被本 PRD §10 继承，命名沿用（"Knowledge Bridge"替代"RAW Link"以更准确表达双向）。

### 3.2 调研② — 外部肩膀（Codex 主笔，深度调研开源生态）

**贡献**：把 ScoutFlow 在开源世界里的位置定位清楚——

- 没有任何一个开源项目完整覆盖 ScoutFlow 的目标
- 但分布在 5 段链路上的成熟项目可以借**思想**（不 fork）：
  - 多源 + H5 + SQLite：MyTube
  - 自动归档 + 浏览器入口：TubeArchivist
  - 源规则订阅模型：Pinchflat
  - B 站同步：bili-sync + BBDown
  - 小红书只读：redbook + xhs-research + ReaJason/xhs
  - ASR：faster-whisper（默认）+ WhisperX（高级）+ Transcription Stream（审校 UI 范式）
  - 知识库参考：Karakeep（卡片 + 标签）+ Khoj（second brain）+ Open WebUI（RAG + transcript 引用）
  - 流程画布：React Flow / ComfyUI / Kestra

**采纳**：本 PRD §22 附录 A 保留外部肩膀对照表。具体技术决定（faster-whisper 默认、SQLite + FTS5、React Flow 做 Pipeline Canvas、Karakeep 风格做 Library 卡片）全部源自调研②。

### 3.3 调研③ — 架构纠偏（Opus 主笔，反驳 console-first）

**贡献**：把"控制台 + 模块工坊"从松散描述钉成硬架构：

- **不是 console-first，是 authority-first**
- Authority 不是单一的 SQLite，是三件套：**`SQLite + FS artifact layout + state words`**
- Workers 必须 stateless（避免常驻进程把单体大中台塞回来）
- thin API 不是"完全没业务"，而是"contract enforcement 四件事"：validation / idempotency / state transition / job receipt
- Step 顺序：**FS layout → state words + schema → Console IA → 模块 contract**

并在用户最终拍板段补充：

- FS 主路径不表达 source 归属（避免 N:N 关系被强行 1:N 化）
- capture-manifest.json 保留 source 谱系（first_seen_source_id / discovery_modes）
- DB 用 `capture_sources` 多对多关系表

**采纳**：本 PRD 的脊梁。§4 / §5 / §6 / §7 / §8 全部按 authority-first 顺序展开。Step 0a 已锁版本完全采纳调研③ 的最终建议。

### 3.4 三份调研的统一裁决

最终 ScoutFlow v1 = **Authority-first 的 Mission Control**：

```

外部肩膀（调研②）：    BBDown / yt-dlp / faster-whisper / WhisperX / SQLite-FTS5 / React Flow / 不 fork 任何项目
```

三份各司其职，没有一份被丢弃，也没有一份被全盘抄袭。

## 4. Authority-first 架构总览

### 4.1 四层堆栈

ScoutFlow v1 的所有模块按 4 层划分，**层间严格遵守"上层只读下层 + 下层无下游知识"**：

```text
┌────────────────────────────────────────────────────────────┐
│  L3  Console (projection layer)                            │
│      H5 前端 / Mission Control / 9 页面                     │
│      唯一读 L0 + 调用 L2 API；从不直接写 L0                   │
├────────────────────────────────────────────────────────────┤
│  L2  Thin API (contract enforcement)                       │
│      FastAPI / 路由层                                       │
│      validation / idempotency / state transition / receipt │
│      不放产品业务逻辑（推荐排序 / 规整规则全在 L1）              │
├────────────────────────────────────────────────────────────┤
│  L1  Workers (stateless functions)                         │
│      Source adapter / Media / ASR / Normalizer /           │
│      Recommender / Reranker / RAW Linker                   │
│      读 L0 view，写 L0 view（通过 L2 API 写）                 │
│      无常驻进程；无内部状态；可并发；可横向重放                  │
├────────────────────────────────────────────────────────────┤
│  L0  Authority (3 件套，唯一权威)                            │
│      • SQLite (data + state + relations)                   │
│      • FS artifact layout (大对象 + evidence)                │
│      • state words contract (lifecycle vocabulary)         │
└────────────────────────────────────────────────────────────┘
```

### 4.2 为什么是 4 层不是 3 层

很多类似系统直接 Console + Workers + DB（3 层）。ScoutFlow 把 thin API 单独列出来，理由是：

- **多 agent 协作需要 contract enforcement 的物理边界**。如果没有 thin API，每个 worker（可能是 Codex 写的 / 可能是 Hermes 触发的 / 可能是 OpenClaw 调度的）都自己直连 SQLite 写状态，state machine 必然漂
- **Console 是 projection layer，不应该有任何写权**。Console 调用 API，API 校验 + 入库。这是 web 应用的常识，但 ScoutFlow 强调因为多 agent 让它成了**生死攸关**的常识
- **Workers 通过 API 写而不是直接 SQL，是 stateless 的物理保证**。worker 不能 "open transaction → 长事务 → commit"，必须 "请求 API → API 校验 → API 落 DB"。这天然让 worker 不能持有 DB 状态

### 4.3 Authority 三件套各司其职

| Authority 件 | 负责 | 不负责 |
|---|---|---|
| **SQLite** | 结构化数据 / 状态 / 关系 / 索引 / 全文检索 / 队列 | 大对象（音频 / 视频 / ASR 原始 JSON / HTML 快照） |
| **FS layout** | 大对象的物理寻址 / evidence 包 / log append-only | 关系表达（多对多 / 状态机） |
| **State words** | 跨 worker / 跨 agent 的语义一致性 | 业务逻辑 / 排序规则 |

三件不可分割：

- 只有 SQLite：path 字段成空头支票（worker 各自拼路径）
- 只有 FS：状态查询要扫目录树（不可扩展）
- 只有 state words：没物理实体支撑

**任何 Worker / Agent 改 ScoutFlow 都必须同时遵守这三件**。

### 4.4 Workers stateless 的物理含义

调研③已经讲过原则。本 PRD 把它锁成可执行规则：

| Worker 行为 | stateless 含义 |
|---|---|
| 启动 | 进程从 0 加载，读必要 capture_id + job_id + params |
| 读 | 只通过 L2 API 读 L0（GET /captures/{id}, GET /captures/{id}/jobs/{type}） |
| 写 | 只通过 L2 API 写 L0（POST .../jobs/{type}/start → /complete） |
| 大对象 | 直接写 FS（按 §5 layout），但路径必须是 API 返回的"约定路径"，worker 不自拼 |
| 中途崩溃 | 重启后从 API 拿到当前 job 状态（running / failed），决定 resume 或 restart |
| 内部状态 | **零内部状态**。任何"上次跑到哪"必须从 L0 读，不许放内存 / 不许放本地 sqlite-of-worker |
| 并发 | worker 可水平扩展，多实例共享 L0；同一 capture_id × job_type 由 API idempotency 保证不重做 |

**反例（禁止）**：

- ❌ worker 进程内开个 dict 缓存"已处理 capture_id"，进程重启就丢
- ❌ worker 自己开 SQLite 连接长事务跨多步操作
- ❌ worker 自己创建目录 / 自己拼 path 不告诉 API
- ❌ 一个 worker 同时持有多个 capture_id 锁

**正例（应该）**：

- ✅ worker 收到 job 请求 → API 返回 capture_id + 约定 path + params → worker 跑完 → POST /complete 携带产物 path 和 metadata → API 校验 → 入库

### 4.5 Thin API 的"thin"边界

| 进 API（contract enforcement） | 不进 API（业务逻辑） |
|---|---|
| validation（payload 校验 / state 合法性） | 推荐分计算公式 |
| idempotency（dedupe key / unique constraint） | 关键词召回规则 |
| state transition enforcement（合法转移 + 触发器） | LLM 重排 prompt |
| job receipt（worker 拿任务 / 报完成） | RAW patch 生成规则 |
| auth（暂无，单机本地） | 摘要 / 论点 / 引用提取 prompt |
| audit logging（每个状态转移留痕） | 标签分类规则 |

**API 不做就交给 Workers 做**。worker 跑完业务逻辑后，把结果 POST 给 API，API 只负责"落 DB + 转 state"。

### 4.6 Console 是 projection layer 的硬约束

Console 必须遵守：

1. **零写权**。Console 不能直接 `INSERT / UPDATE / DELETE` SQLite，不能直接写 FS 大对象
2. **所有写操作都走 L2 API**。包括用户在 UI 上点【采集】也是发 POST，不是前端直接写
3. **Console 可以缓存 L0 数据用于渲染**，但缓存失效时必须重读 L0，不能展示陈旧数据当作权威
4. **Console 不参与 state transition 决策**。UI 不要写"如果 status==X 就改成 Y"的逻辑，那是 API 的职责
5. **Console 失败 / 离线 / 不打开**，整个 ScoutFlow 仍可运行（Workers + Hermes + OpenClaw 通过 API 协作即可）

这条最后一点很关键：**Console 是可选投影，不是必需中枢**。

### 4.7 多 agent 协作的物理边界

四层堆栈对多 agent 的友好度，体现在以下不变量：

| 不变量 | 物理实现 |
|---|---|
| Codex 写一个 worker 不会破坏 Opus 写的另一个 worker | 两个 worker 独立 API 调用，不共享内存 / 不共享磁盘文件（除 L0） |
| Hermes 定时任务跑挂不影响 user 在 Console 的操作 | Hermes 通过 API 写 candidate，user 通过 API 操作主线，互不阻塞 |
| OpenClaw 改一个 capture record 不会让整个 DB 锁死 | API 只持有短事务，worker 跑长任务时 DB 是释放状态 |
| ChatGPT Pro 给的调研报告进入候选区不污染主线 | candidate 文件落到 `candidates/chatgpt-pro/`，user 拍板才合并到 decisions.md |
| 任何 agent 中途下线，其他 agent 继续工作 | stateless workers + L0 唯一权威，无 agent 持有"独家状态" |

| LLM 规整 jsonl（claims / quotes） | ✅ | ❌ | 关键 claim 进 `normalized_docs.claims_json`（≤10KB） |
| log / receipt / stderr | ✅ | ❌ | — |
| source raw API response | ✅ | ❌ | — |
| source HTML snapshot | ✅ | ❌ | — |
| 索引字段（title / creator / duration / publish_at） | ❌ | ✅ | — |
| 状态字段（status / promotion_state） | ❌ | ✅ | — |

### 5.7 capture-manifest.json schema

每个 capture 创建时由 API 写入，schema 如下：

```json
{
  "schema_version": "1",
  "capture_id": "01HXXXXX0YQXXXXXX0YQXXXXXX",
  "platform": "bilibili",
  "platform_item_id": "BV1xxxxxxxxx",
  "canonical_url": "https://www.bilibili.com/video/BV1xxxxxxxxx",
  "title": "原始平台标题（不修改）",
  "creator": {
    "platform_creator_id": "1234567",
    "name": "影视飓风",
    "handle": "@yingshijufeng",
    "profile_url": "https://space.bilibili.com/1234567"
  },
  "duration_seconds": 845,
  "published_at": "2026-04-29T08:00:00+08:00",
  "discovered_at": "2026-05-02T10:30:00+08:00",
  "discovery": {
    "first_seen_source_id": "src_0007",
    "first_seen_mode": "account_subscription",
    "discovery_modes": ["account_subscription", "keyword_match"],
    "source_hit_count": 2
  },
  "capture_mode": "audio",
  "created_by": "hermes",
  "rights_note": "fair-use research / non-redistribution / personal study",
  "checksums": {
    "raw_api_response": "sha256:abc123..."
  }
}
```

**字段说明**：

- `schema_version`：manifest schema 版本号。任何字段增删要 bump
- `capture_id`：与目录名一致
- `platform_item_id`：平台原生 ID（BV / 小红书 note ID）
- `canonical_url`：规范化 URL（去除追踪参数）
- `title` / `creator`：发现时的快照值，平台后续改名不追溯
- `duration_seconds`：作品时长（秒）
- `published_at`：作品发布时间（带时区）
- `discovered_at`：ScoutFlow 第一次发现的时间
- `discovery`：发现谱系
  - `first_seen_source_id`：第一次命中这条 capture 的 source（不可变）
  - `first_seen_mode`：第一次命中的发现模式
  - `discovery_modes`：所有命中模式的并集（随时间增长）
  - `source_hit_count`：被多少个 source 命中（DB `capture_sources` count 的镜像）
- `capture_mode`：用户选择的采集深度（`metadata_only` / `audio` / `video` / `transcript_only`）
- `created_by`：触发创建的 actor（`user` / `hermes` / `openclaw` / `codex` / `chatgpt-pro`）
- `rights_note`：版权备注（默认 fair-use research）
- `checksums`：关键 evidence 文件的 sha256，便于检测篡改

**变更规则**：

- `schema_version` / `capture_id` / `platform` / `platform_item_id`：永不变
- `discovered_at` / `first_seen_source_id` / `first_seen_mode`：永不变
- `discovery_modes` / `source_hit_count`：随后续 source 命中追加
- `capture_mode`：可被 user 升级（`metadata_only` → `audio` → `video`），降级 reject
- 其他字段：从平台原始响应快照写入，不主动刷新（除非 manual re-fetch）

### 5.8 懒创建规则（物理生命周期）

6 区是**逻辑必须**（DB 里都有对应 path 列），但**物理懒创建**：

| 时机 | 物理创建 | 何时填充 |
|---|---|---|
| `POST /captures/discover` | `<platform>/<capture_id>/bundle/` 目录 + `capture-manifest.json` | 同步创建 |
| `metadata-fetch` worker 跑完 | `bundle/raw-api-response.json` | metadata-fetch 完成 |
| `media-download` worker 跑完 | `media/` 目录 + 实际文件 | media-download 完成 |
| `audio-extract` worker 跑完 | `media/audio.wav`（如视频已有则跳过） | audio-extract 完成 |
| `asr` worker 跑完 | `transcript/` 目录 + `raw.json` | asr 完成 |
| `asr-postprocess` worker 跑完 | `transcript/segments.jsonl` + `media/derived/audio-segments/` | postprocess 完成 |
| `normalize` worker 跑完 | `normalized/` 目录 + 各 jsonl/md | normalize 完成 |
| `raw-linker` worker 跑完 | `links/` 目录 + 各 jsonl | raw-linker 完成 |
| 任意 worker 跑过 | `logs/jobs/{job_type}-{ts}.log` + `logs/receipts/{job_id}.receipt.json` | worker 跑完 |

**好处**：

- `ls capture_id/` 一眼能看出"这条 capture 走到哪步"，不需要查 DB
- 失败的 capture 不会创建空目录留下幻影
- 备份时可以按目录存在性判断完整度

**约束**：

- worker 不允许"提前创建"未来步骤的目录（如 asr worker 不允许创建 `normalized/`）
- 如果 worker 失败但已创建目录，必须保留（用于 debug），不要清理
- API 层在 capture 删除时才清理整个 `<platform>/<capture_id>/`

### 5.9 跨 worker 写边界（隔离规则）

| Worker | 允许写 | 禁止写 |
|---|---|---|
| source-adapter（discover） | API 调用 → 触发 API 写 manifest | 直接写 FS |
| metadata-fetch | `bundle/raw-api-response.json` / `bundle/source-page-snapshot.*` | 其他区 |
| media-download | `media/video.*` / `media/audio.*` / `media/thumbnail.*` / `media/subtitle.*` / `media/danmaku.*` | 其他区 / `media/derived/` |
| audio-extract | `media/audio.*`（仅当不存在时） | 其他 |
| asr | `transcript/raw.json` | 其他 |
| asr-postprocess | `transcript/segments.jsonl` / `media/derived/audio-segments/` / `media/derived/waveform.json` | 其他 |
| normalize | `normalized/*` | 其他 |
| raw-linker | `links/*` | 其他 |
| 所有 worker | 自己负责的 `logs/jobs/*` + `logs/receipts/*`（通过 API） + `logs/errors/*`（仅失败时） | 别人的 logs |
| API（仅写 manifest） | `bundle/capture-manifest.json` 字段追加 | media / transcript 等区 |
| Console | **任何 FS 都不能直接写**，必须走 API | 全部 |

**违反 = reject**。代码 review 必须 grep "open(.*'w'" 等可疑写操作，确认每个写都符合上表。

### 5.11 错误恢复与一致性

#### 5.11.1 worker crash

worker 跑到一半进程被杀：

- 已写完的文件保留（可能是部分文件）
- DB job 状态停留在 `running`
- 下次 worker 重启时通过 API 拉 stale running jobs，按以下策略处理：
  - 检查 receipt 是否存在 → 不存在则视为未完成 → reset 到 `queued` 重跑
  - 检查关键产物文件是否存在 → 不存在则 reset

#### 5.11.2 FS 与 DB 不一致

如果 DB 说 `transcript_documents` 存在但 FS 没有 `transcript/raw.json`：

- API 返回 503（数据完整性错误）
- Console 显示该 capture 为 `inconsistent` 状态
- 需要 user 手动选择：[重跑 ASR] / [删除 transcript 记录]

#### 5.11.3 孤儿 FS（DB 没记录但 FS 有目录）

定期 (每周) 跑 `scoutflow.checkfs` 巡检：

- scan `data/artifacts/<platform>/<capture_id>/`
- 比对 DB `captures.id`
- 孤儿目录列表写到 `data/orphans-{date}.jsonl`
- user 决定：[导回 DB] / [删除目录]

### 5.12 Step 0a 锁定后不可变约束

以下条目**不再讨论**，直到 amendment 流程：

- ✅ FS 主路径 = `artifacts/<platform>/<capture_id>/`，无 source_id
- ✅ 6 区分层（bundle / media / transcript / normalized / links / logs）
- ✅ capture_id = ULID
- ✅ 文件名固定语义（见 §5.5.4 表）
- ✅ capture-manifest.json schema v1（见 §5.7）
- ✅ DB `capture_sources` 表负责多对多关系
- ✅ 懒创建（API 在 discover 时只建 bundle/ + manifest）
- ✅ 跨 worker 写隔离（见 §5.9 表）

任何 agent / worker / spec 引用 FS 路径都必须按本章。

## 6. Step 0b — State Words & Lifecycle（草案 v1，待 user 拍板）

### 6.1 拍板状态

- **状态**：DRAFT v1
- **起草人**：Opus
- **下一步**：user 审 + Codex 反审 + 三方拍板锁定

### 6.2 为什么 state words 是 authority 三件套之一

调研③ 已经讲清楚原则。本章把它落到实操：

- 每个 capture 在生命周期里经过 N 个状态
- 每个 worker 跑前读一种状态、跑完写另一种状态
- 多 agent / 多 worker 必须共享同一套状态词汇（不能 worker A 用 `transcribed` worker B 用 `asr_done`）
- state words 是跨 worker 跨 agent 跨 session 的语义契约

### 6.3 capture_records.status — 主生命周期

#### 6.3.1 完整状态枚举（17 个 + 7 个失败态）

```text
正常路径（按时间顺序）：
  discovered            刚发现，capture_id 已生成，仅有 manifest
  metadata_fetched      平台 API 响应已落 bundle/raw-api-response.json
  queued                已入 capture queue 等待 download
  media_downloading     正在下载视频/音频
  media_ready           视频/音频已就位
  audio_extracting      正在从 video 提取 audio
  audio_ready           audio 已就位（无论从 download 还是 extract）
  transcribing          ASR 正在跑
  transcript_ready      transcript/raw.json 已就位
  normalizing           LLM 正在规整
  doc_ready             normalized/* 已就位
  indexing              FTS / vector 正在建索引
  indexed               索引已建
  linking               raw-linker 正在跑
  linked                links/* 已就位
  archived              user 标记已用 / 不再处理
  superseded            被同一 platform_item_id 的更新版替代

失败路径：
  failed_metadata       metadata-fetch 失败
  failed_download       media-download 失败
  failed_audio          audio-extract 失败
  failed_asr            ASR 失败
  failed_normalize      LLM 规整失败
  failed_index          索引失败
  failed_link           raw-linker 失败
```

#### 6.3.2 合法状态转移图

```text
                discovered
                    ↓ metadata-fetch worker
                metadata_fetched
                    ↓ user / scheduler queue
                queued
                    ↓ media-download worker
        ┌─ failed_download ←──→ media_downloading
        ↓                           ↓
        retry              media_ready
                                    ↓ audio-extract worker (跳过 if audio 已存在)
                            audio_extracting
                                    ↓
                            audio_ready
                                    ↓ asr worker
                            transcribing ←──→ failed_asr
                                    ↓                     ↓
                            transcript_ready          retry
                                    ↓ normalize worker
                            normalizing ←──→ failed_normalize
                                    ↓                     ↓
                            doc_ready              retry
                                    ↓ index worker
                            indexing ←──→ failed_index
                                    ↓             ↓
                            indexed         retry
                                    ↓ raw-linker worker
                            linking ←──→ failed_link
                                    ↓             ↓
                            linked          retry
                                    ↓ user action
                            archived  /  superseded
```

#### 6.3.3 转移触发器

| 转移 | 谁触发 | 触发条件 |
|---|---|---|
| `discovered → metadata_fetched` | metadata-fetch worker complete | API 响应 200 + raw-api-response.json 已写 |
| `metadata_fetched → queued` | user / scheduler | user 在 Source Radar 点【采集】，或 Hermes 自动入队 |
| `queued → media_downloading` | media-download worker start | worker 拿到 job |
| `media_downloading → media_ready` | media-download worker complete | 文件 sha256 匹配预期 |
| `media_ready → audio_extracting` | audio-extract worker start | capture_mode 包含 audio 且 audio.* 不存在 |
| `audio_extracting → audio_ready` | audio-extract worker complete | audio.wav 已写 + 时长校验通过 |
| `audio_ready → transcribing` | asr worker start | worker 拿到 job |
| `transcribing → transcript_ready` | asr worker complete | transcript/raw.json 已写 |
| `transcript_ready → normalizing` | normalize worker start | worker 拿到 job |
| `normalizing → doc_ready` | normalize worker complete | normalized/* 已写 |
| `doc_ready → indexing` | index worker start | worker 拿到 job |
| `indexing → indexed` | index worker complete | FTS 已索引 |
| `indexed → linking` | raw-linker worker start | worker 拿到 job |
| `linking → linked` | raw-linker worker complete | links/* 已写 |
| `linked → archived` | user action | user 在 Library 标记【已用】或【归档】 |
| `* → superseded` | metadata-fetch worker | 发现同 platform_item_id 但 published_at 更新 |
| `*ing → failed_*` | worker error | 抛异常或 receipt 标失败 |
| `failed_* → queued` | user / scheduler | retry 操作 |

#### 6.3.4 跳过规则

某些 capture 不需要走完整链路：

- `capture_mode = metadata_only`：链路终止于 `metadata_fetched`
- `capture_mode = transcript_only` + 平台已有官方字幕：跳过 ASR，从 subtitle 直接进 `transcript_ready`
- 单纯入库不要 RAW 关联：跳过 `linking`，从 `indexed` 直接进 `linked`（标 `linking_skipped`）

### 6.4 jobs.status — 任务级状态

每条 capture 在生命周期里产生 N 个 jobs（一个 worker 跑一次 = 一个 job）。jobs 有自己的状态：

```text
queued        刚入队，等 worker 拉
running       worker 已拉走开始跑
succeeded     完成 + receipt 写入 + 状态转移成功
failed        worker 报错或 receipt 写失败
skipped       条件不满足跳过（如 audio 已存在不需要 extract）
canceled      user 主动取消
```

合法转移：
```text
queued → running (worker 拉) → succeeded / failed / canceled
queued → skipped (前置条件检查)
running → canceled (user 取消)
failed → queued (retry)
```

### 6.5 candidates.status — 多 agent candidate 状态

```text
submitted      agent 提交了候选（候选文件已落 candidates/<agent>/）
reviewing      合并仲裁者（默认 Codex）正在审
needs_user     仲裁者标"需 user 拍板"
accepted       被采纳合并到主线
rejected       被拒绝
```

合法转移：
```text
submitted → reviewing → accepted / rejected / needs_user
needs_user → accepted / rejected
```

### 6.6 source.status — source 自身状态

```text
ready          正常运行
candidate      创建中或待验证（如关键词刚加未确认有结果）
blocked        因登录 / Cookie 失效 / 风控被阻塞
manual         需要人工操作（如小红书需 user 手动 Cookie 刷新）
disabled       user 主动停用
```

### 6.7 promotion_state — 跨 lifecycle 的资产 promotion 维度

继承 codex-metacognition §1.2 的 promotion gate：

```text
L1-only        本 capture 仅用于本次个人研究（默认）
L2-validated   user 标记为有价值，可进入 RAW patch 候选池
L-global       user 标记为高质量，可作为长期参考资产（生成 reusable 引用）
```

promotion_state **不会回退**。一旦从 L1 升 L2，不能再降回 L1（除非整个 capture 删除）。

### 6.8 状态词反例（禁止）

以下命名方式 **禁止** 在 ScoutFlow 任何 worker / spec / API 出现：

- ❌ `done`（不明确 = 哪一步 done）
- ❌ `processing`（不明确 = 哪一步 processing）
- ❌ `error`（用 `failed_*` 标明失败位置）
- ❌ `pending`（用 `queued` 表队列等待 / 用 `discovered` 表初始）
- ❌ `complete`（用 `*_ready` 表示某资产就位）
- ❌ `success` / `ok`（用 `succeeded`）

任何 PR / candidate 中出现以上模糊词必须 reject + amendment。

### 6.9 状态查询惯例

API / Console 查询状态时统一用：

- 单 capture 当前态：`captures.current_status`（== capture_records.status 最新值）
- 单 capture 历史：`SELECT * FROM capture_state_history WHERE capture_id=?` 按时间升序
- 批量按状态过滤：`SELECT * FROM captures WHERE status IN ('queued','media_downloading')`
- 统计仪表盘：`SELECT status, COUNT(*) FROM captures GROUP BY status`

详细 schema 见 §7。

### 7.8 schema 设计原则总结

- **TEXT 主键**：用 ULID，便于跨表 join 不依赖 INTEGER autoincrement
- **路径全用相对路径**：从 ScoutFlow 项目根算起，不写绝对路径（可移植）
- **时间戳全 ISO 8601**：`strftime('%Y-%m-%dT%H:%M:%fZ','now')`
- **enum 全用 CHECK 约束**：违反 enum 直接 INSERT 失败，比应用层校验稳
- **大对象 path 字段**：永远是相对路径字符串，DB 不存大对象本身
- **多对多用关系表**：永远不在主表用 JSON 数组表达多对多
- **append-only 表**（state_history / job_events）：UPDATE 禁止，只 INSERT
- **status enum 与 state words contract 强同步**：§6 改 → §7 改 → migration 改

## 8. Thin API Contract（草案 v1）

### 8.1 拍板状态

- **状态**：DRAFT v1
- **依赖**：Step 0a layout / Step 0b state words + schema
- **下一步**：Step 0b 锁定后同步锁定本章

### 8.2 API 设计原则

- **REST 风格**，资源 + 动作语义清晰
- **JSON-only**（请求和响应都 JSON，不支持 form / xml）
- **路径以资源为中心**（`/captures/{id}/jobs/{type}`，不是 `/run-asr`）
- **状态转移用 PATCH 单字段**（不允许 PUT 全量替换）
- **idempotency-key 必填**（写操作）
- **error 用 problem+json**（RFC 7807 风格）
- **本地单机无 auth**（Phase 1 不做 token / OAuth），但保留 `X-Agent-Id` header 做 audit log

### 8.3 路由总表

```text
POST   /jobs/{id}/retry                  reset failed job 到 queued

============ Media Assets ============

============ Transcripts ============

PATCH  /captures/{id}/transcript         user 标记 reviewed=1
POST   /captures/{id}/segments/{seg_id}/tag   user 给 segment 打标（segment_type / user_tag）
PATCH  /captures/{id}/segments/{seg_id}        user 修改 segment text

============ Normalized ============
GET    /captures/{id}/normalized         normalized_docs 详情

============ Knowledge Links ============

PATCH  /links/{id}/accept                user 接受
```

### 8.4 写操作的契约四件事

#### 8.4.1 Validation

- 所有 POST/PATCH body 必须通过 JSON Schema 校验（用 pydantic v2 model）
- enum 字段不通过 → 422
- 外键引用不存在 → 404 with detail
- 路径参数和 body 字段冲突 → 422

#### 8.4.2 Idempotency

写操作必须支持以下 idempotency key 之一：

| 操作 | dedupe key |
|---|---|
| `POST /sources` | hash(platform, source_type, config_canonical_json) |
| `POST /captures/discover` | (platform, platform_item_id) |
| `POST /captures/{id}/jobs/{type}` | (capture_id, job_type) — 已 succeeded 返 409 + 既有 job_id |
| `POST /captures/{id}/sources` | (capture_id, source_id, hit_mode) |
| `POST /candidates` | (agent_id, file_path) |
| `POST /captures/{id}/links` | batch upsert by (capture_id, target_kind, target_path, relation_type) |

请求方可在 Header 加 `Idempotency-Key: <ulid>`，API 缓存 24 小时内同 key 的响应。

#### 8.4.3 State Transition Enforcement

非法状态转移直接 409。例如：

- `captures.status` 当前 `archived`，POST `/captures/{id}/restart-pipeline` → 409
- `jobs.status` 当前 `succeeded`，POST `/jobs/{id}/start` → 409
- `captures.status` 当前 `discovered`，worker POST `/jobs/{id}/complete` 标 `media_ready` → 409（未经过 `media_downloading`）

API 内部维护转移图（§6 的合法转移），任何越级 reject。

#### 8.4.4 Job Receipt

worker 完成 job 必须 POST `/jobs/{id}/complete` 携带：

```json
{
  "produced_assets": [
    {
      "asset_type": "audio",
      "file_path": "data/artifacts/bilibili/01HXX.../media/audio.wav",
      "sha256": "abc123...",
      "size_bytes": 12345678,
      "duration_seconds": 845.0,
      "sample_rate": 16000,
      "channels": 1
    }
  ],
  "engine": "BBDown",
  "engine_version": "1.6.2",
  "engine_logs_path": "data/artifacts/bilibili/01HXX.../logs/jobs/media-download-2026-05-02T10:30Z.log",
  "duration_seconds": 12.5,
  "next_state_hint": "media_ready"
}
```

API 校验 sha256 + 文件实际存在 + 大小匹配，全过才入库 + 落 receipt + 推进 state。

任何字段不匹配 → API 拒绝 + worker 标失败。

### 8.5 Error 响应格式（problem+json）

```json
{
  "type": "https://scoutflow.local/errors/invalid-state-transition",
  "title": "Invalid state transition",
  "status": 409,
  "detail": "Cannot transition capture 01HXX from 'archived' to 'queued'. Use POST /captures/{id}/restart-pipeline to revive.",
  "instance": "/captures/01HXX/restart-pipeline",
  "context": {
    "current_state": "archived",
    "requested_state": "queued"
  }
}
```

错误码命名空间：

- `4xx`: client 错误（payload / 状态非法 / 缺资源）
- `5xx`: server 错误（DB 错 / FS 错 / 外部依赖）
- `409`: idempotency 冲突 + state transition 非法
- `422`: validation 失败
- `503`: FS / DB 不一致

### 8.6 限流与并发

- API 单机本地，不做请求级限流
- worker 拉任务并发上限：默认 2，可配置（避免本机带宽爆）
- 单 capture × 单 job_type 并发：API 强制 1（idempotency 保护）

### 8.7 Audit Logging

每个写操作 API 自动写 `capture_state_history` / `job_events` / `audit_log`（一张未来加的表）。

audit log 字段：

```json
{
  "timestamp": "2026-05-02T10:30:00.123Z",
  "actor": "user",  // header X-Agent-Id 或 user
  "method": "POST",
  "path": "/captures/01HXX/archive",
  "request_summary": "{...截断 1KB...}",

  "duration_ms": 12
```

## 9. Workers 模块设计

### 9.1 总览

ScoutFlow 一期共 7 类 worker（按 lifecycle 顺序）：

| # | Worker | 输入 state | 输出 state | 引擎 | 主要写区 |
|---|---|---|---|---|---|
| 1 | source-adapter | (source 触发) | discovered | 平台 SDK | bundle/raw-api-response.json |
| 2 | metadata-fetch | discovered | metadata_fetched | yt-dlp / BBDown / xhs | bundle/* |
| 3 | media-download | queued | media_ready | BBDown / yt-dlp | media/video/audio/thumbnail/... |
| 4 | audio-extract | media_ready | audio_ready | ffmpeg | media/audio.wav（仅当不存在） |
| 5 | asr | audio_ready | transcript_ready | faster-whisper | transcript/raw.json |
| 6 | asr-postprocess | transcript_ready | （隐式，扩展 transcript_ready） | 内部 Python | transcript/segments.jsonl, media/derived/* |
| 7 | normalize | transcript_ready | doc_ready | LLM (claude / gpt) | normalized/* |
| 8 | index | doc_ready | indexed | SQLite FTS5 | （无 FS，写 DB） |
| 9 | raw-linker | indexed | linked | LLM + RAW scanner | links/* |

加 2 类后台 worker：

| # | Worker | 触发 | 作用 |
|---|---|---|---|
| 10 | recommender | 定时 + on source-item 创建 | 计算召回分入 recommendations |
| 11 | reranker | 定时 batch | LLM 重排 + 写理由 |

### 9.2 通用 worker 协议

每个 worker 必须遵守：

#### 9.2.1 拉任务

```python
job = api.post(f"/captures/{capture_id}/jobs/{job_type}", body={...})
# 或定时任务模式：
jobs = api.get("/jobs?status=queued&job_type=asr&limit=1")
api.post(f"/jobs/{jobs[0].id}/start")
```

#### 9.2.2 工作

worker 内部按职责跑实际工作。期间不能持有 DB 连接（API 已经放手了）。worker 可以：

- 直接写 FS（按约定 path）
- 调用外部进程（ffmpeg / BBDown）
- 调用 LLM API
- 在内存里计算

不能：

- 直接连 SQLite
- 改其他 worker 的 FS 文件
- 创建未约定的 path

#### 9.2.3 报告完成

```python
api.post(f"/jobs/{job_id}/complete", body={
    "produced_assets": [...],
    "engine": "...",
    "engine_version": "...",
    "engine_logs_path": "...",
    "duration_seconds": ...,
    "next_state_hint": "..."
})
```

#### 9.2.4 报告失败

```python
api.post(f"/jobs/{job_id}/fail", body={
    "error_class": "DownloadTimeout",
    "error_message": "BBDown returned timeout after 120s",
    "stderr_path": "logs/errors/{job_id}.stderr"
})
```

### 9.3 Source Adapters

#### 9.3.1 bili-adapter

**职责**：扫描 source（账号 / 关键词 / 链接 / 列表），产出 source_items；接受单 capture 创建。

**底层工具**：

- 元数据：BBDown info 命令 / 直接调用 B 站 API
- 媒体下载：BBDown
- 凭据：本机已有的 BBDown cookie

**接口**：

```python
def scan_account(source: Source) -> list[SourceItem]:
    """扫一个 UP 主新发布作品"""

def scan_keyword(source: Source) -> list[SourceItem]:
    """关键词搜索"""

def scan_list(source: Source) -> list[SourceItem]:
    """收藏夹 / 合集 / UP 主投稿"""

def fetch_metadata(capture_id: str, platform_item_id: str) -> Manifest:
    """单 capture 的 metadata fetch"""
```

**风险/限制**：

- B 站 cookie 失效 → source.status='blocked'，user 必须手动刷新
- B 站风控（频繁请求被限流）→ 自动降级到 polling_interval × 2，连续 3 次失败 → blocked
- 4K / 大会员视频：尊重账号权限，不绕过

#### 9.3.2 xhs-adapter

**职责**：扫小红书（博主 / 关键词 / 笔记 URL / 收藏）。

**底层工具**：

- xhs-research skill（已有）+ ReaJason/xhs Python 库
- redbook CLI 作为 fallback

**接口**：与 bili-adapter 同形态，但 capability_level 默认 `manual`（首版只读 + 频率限制）。

**风险/限制**：

- 小红书反爬严格 → 默认 polling_interval ≥ 6 小时
- 仅 metadata + 笔记正文（图片）；视频笔记的视频文件为可选下载
- 不实现：评论 / 点赞 / 关注 / 发布
- 一次性最多 20 条，超出强制分批

#### 9.3.3 youtube-adapter（Phase 2）

**职责**：YouTube 单条 / 频道 / playlist。

**底层工具**：yt-dlp。

**Phase 1 不实现**。

### 9.4 Media Worker

**职责**：

- media-download：从平台下载视频 / 音频 / 缩略图 / 字幕 / 弹幕
- audio-extract：从 video 提取 audio.wav（16kHz mono，给 ASR 用）
- derive-preview / derive-waveform：派生产物

**关键决定**：

- 默认 `capture_mode=audio` 时，直接下载音频（BBDown / yt-dlp 都支持），不下视频
- 当平台不支持纯音频流，下载视频后立即 audio-extract，extract 完可选删除 video 或保留（按 source.config）
- audio.wav 标准化：16kHz / mono / pcm_s16le，便于 ASR

### 9.5 ASR Worker

#### 9.5.1 faster-whisper（默认）

- 模型：`large-v3`（默认）/ `medium`（快速）
- GPU：Mac M5 Apple Silicon（CTranslate2 CPU + Metal）
- 输出：`transcript/raw.json`，含每个 segment 的 start/end + 词级时间戳

#### 9.5.2 WhisperX（高级模式）

- 加 word-level alignment + speaker diarization
- 仅在 capture.metadata 中标 `enable_diarization=true` 时使用
- Phase 1 默认关闭，访谈类内容可手动开启

#### 9.5.3 字幕优先策略

- 如果 `media/subtitle.*` 已存在（平台原生字幕）且语言匹配，跳过 ASR 直接 `transcript_ready`
- 字幕走 `subtitle-to-transcript` 内部小 worker 转 raw.json 格式
- 避免重复跑 ASR

### 9.6 ASR Postprocess

**职责**：

- raw.json → segments.jsonl（按 punctuation / silence 重新切段）
- 切对应 audio segments（`media/derived/audio-segments/seg_NNNN.wav`）
- 生成 waveform.json
- 给每段一个 `segment_index`

**为什么不直接合并到 ASR**：保持 ASR 纯粹（输入音频，输出转写），后处理是独立单元，便于换 ASR 引擎不影响后处理。

### 9.7 Normalizer (LLM)

**职责**：transcript_ready → 规整为可消费 markdown + 元数据。

**LLM 路由**（按任务复杂度）：

- 三句话摘要：claude-haiku-4-5 / gpt-4-mini（便宜）
- 论点 / 引用 / 选题候选：claude-sonnet-4-6（默认）
- 长文规整 / 跨段关联：claude-opus-4-7（仅大于 30 分钟视频）

**输出文件**：

- `normalized/summary.md` — 三句话 + tags
- `normalized/claims.jsonl` — 一行一论点
- `normalized/quotes.jsonl` — 一行一引用 + segment_id 引用
- `normalized/topic-candidates.jsonl` — 选题候选
- `normalized/structured.md` — 完整结构化文档

**Prompt 框架**（草案，Phase 1 完善）：

```text
[system]
你是 ScoutFlow 的 normalizer。任务是把 ASR 转写规整为可引用的素材。
要求：
- 三句话摘要必须 ≤ 60 字
- 论点必须可独立引用，不依赖上下文
- 引用必须给出原文 + segment_id
- 选题候选要给出"为什么这个选题值得做"的 1 句理由
- 不要发明事实，全部基于转写

[user]
平台: {platform}
作者: {creator_name}
标题: {title}
转写:
{transcript_full_text}

输出格式（JSON Lines）：
...
```

### 9.10 Worker 部署

Phase 1 单机：

- 所有 worker 都是 Python 进程
- 一个 master scheduler（内部 daemon）每 5 秒拉 `v_jobs_queue` 分发到对应 worker
- worker 进程池：`scoutflow-worker --type asr --concurrency 1` 等
- 失败 capture 不阻塞其他 capture

Phase 4 多 agent 时：

- Hermes / OpenClaw 作为外部 trigger（POST /sources/{id}/scan）
- worker 仍然 stateless，可以由任意 agent 触发

### 9.11 Worker 错误分类

| 错误类 | 含义 | 处理 |
|---|---|---|
| `NetworkTimeout` | 平台请求超时 | 重试 3 次，指数 backoff |
| `AuthExpired` | cookie 失效 | source.status='blocked'，alert user |
| `RateLimit` | 平台限流 | 暂停该 source 1 小时 |
| `MediaCorrupt` | 下载文件 sha 不匹配 | 重新下载 1 次，再失败 = failed |
| `ASRFailed` | whisper 内部错 | 重试 1 次，失败 = failed |
| `LLMQuotaExceeded` | LLM API 超额 | 暂停 normalize worker，alert user |
| `FSWriteFailed` | 磁盘写失败 | 立即 failed，alert（可能磁盘满） |
| `SchemaViolation` | API 拒收 receipt | 立即 failed（不重试，因为产物有问题） |

## 11. Recommendation Engine

### 11.1 设计目标

把"4 类入口全上"的多源信号转化为有理由的推荐，避免：

- 只按时间排序的信息流（杂、无优先级）
- 只按热度排序（热点不一定相关）
- 黑盒推荐（user 不知道为什么推这条）

### 11.2 两阶段：召回 + 重排

```text
所有 source_items
       ↓ 规则召回（recommender worker）
[召回池: ~50-200 条/天]
       ↓ LLM 重排（reranker worker）
[Source Radar 显示: top 20-30]
```

### 11.3 召回阶段 — 规则分

每条 source_item 计算召回分（0-1）。规则：

| 信号 | 权重 | 算法 |
|---|---|---|
| 来源是账号订阅 | +0.3 | 命中即加 |
| 来源是关键词命中 | +0.2 | 命中即加 |
| 来源是列表 | +0.1 | 命中即加 |
| 来源是单条链接 | +0.5 | user 主动 |
| 多 source 命中（N≥2） | +0.1 × (N-1) | 叠加 |
| 创作者在 RAW 中提及 | +0.2 | match RAW node creators |
| 标题命中关键词 | +0.1 / 词 | 关键词列表 |
| 时长在偏好区间（5-30 分） | +0.05 | |
| 时长 < 1 分（短视频） | -0.1 | |
| 互动数 > 阈值 | +0.05 | |
| 发布时间 < 7 天 | +0.05 | |
| 发布时间 > 90 天 | -0.05 | |

最终召回分 = sum(命中规则的权重)，clipped to [0, 1]。

### 11.4 重排阶段 — LLM + RAW Context

对召回分 ≥ 0.4 的 item 跑 LLM 重排：

```text
[system]
你是 ScoutFlow 的 reranker。给每个候选 1 个 [0,1] 分和 ≤30 字理由。
评分维度：
- 与 user 当前 RAW 知识的相关性
- 内容新颖度（user 是否看过类似）
- 论证质量预估（标题党 / 干货）
- user 当前阶段是否需要这类素材

[user]
RAW 高频节点（最近 30 天）:
- AI Agent 选型
- 内容生产工作流
- ...

候选:
1. 影视飓风《AI 工具实测》| Bilibili | 14:25
2. ...

输出: JSON Lines，每行 {item_index, score, reason, suggested_action}
```

### 11.5 心动度衰减（Opus 提的元问题落地）

每条 recommendation 的 `score_excitement` 维度：

- 创建时初始 score（来自 LLM）
- 每天衰减 5%
- user 看过（status=seen）但 7 天未操作 → 衰减加速
- user 操作（accept / capture）→ 心动度归零（不再推同一条）
- user 反复 ignore 同类（同 creator / 同主题）→ 抑制因子加到 reranker prompt

公式：

```text
score_excitement(t) = score_initial × (0.95 ^ days_since_seen) × seen_penalty × ignore_penalty
```

最终 score = 0.5 × score_recall + 0.3 × score_rerank + 0.2 × score_excitement

### 11.6 推荐理由生成

每条推荐必须有可读理由（≤30 字），格式：

```text
"与 RAW 『AI Agent 选型』高度相关 + 多源命中 (账号 + 关键词)"
"短时长视频，符合你最近偏好；可能补充『工具评测』节点"
"作者在 RAW 中提及过；新发表 < 24h"
```

### 11.7 推荐质量监控

- `dashboard/recommendation-quality`：accepted_rate / ignored_rate / hit-after-N-days
- 每月统计：哪些规则权重需要调（如 user 反复 ignore "时长 < 1 分" 加分项 → 降权）
- 不做 A/B 测试（单 user，无对照）

```text
LLM 提取概念 → 匹配 RAW 节点 → 生成 patch 候选
   ↓ links/raw-suggestions.jsonl + DB knowledge_links

   ↓ accept

   ↓ user 在 RAW 自己的工作流里 review + 合并
```

#### 12.2.2 RAW → capture (Reverse Discovery)

```text
user 在 RAW 标缺口 / 在 ScoutFlow 主动触发
   ↓ Knowledge Bridge → Reverse Discovery
LLM 把 RAW 缺口转关键词
   ↓
Source Radar 一次性扫描候选
   ↓ user 选择
Capture Pipeline
```

### 12.3 relation_type 完整枚举

```text
supports         证据支持 RAW 节点的论点
contradicts      与 RAW 节点矛盾，值得辩证
extends          扩展 RAW 节点（新维度 / 新案例）
example_of       是 RAW 节点的具体例子
quote_for        可作为 RAW 节点的引用
definition_of    给 RAW 节点的术语下定义
trend_signal     表明趋势变化
topic_seed       可衍生 DiloFlow 选题
script_material  可作 DiloFlow 台本素材
reverse_discovery 是为了填 RAW 缺口而抓的
```

## 14. 安全与合规边界

### 14.1 平台 ToS 边界

| 平台 | 可做 | 不做 |
|---|---|---|
| Bilibili | 个人研究采集 / 合理频率 / 尊重大会员限制 / 字幕弹幕保留 | 4K 绕过 / 大会员内容绕过 / 高频批量 / 公开再分发 |
| 小红书 | 公开内容只读 / 笔记 / 关注 / 收藏 | 评论 / 点赞 / 关注 / 发布 / 高频 / 视频笔记不抓视频本体 |
| YouTube（Phase 2） | 公开视频 / 字幕 / 元数据 | 付费 / 会员限定 / 区域绕过 |

### 14.2 内容版权红线

- 所有采集内容**仅供个人研究**（rights_note 默认）
- 不公开 ScoutFlow 数据库 / artifact 目录
- 不在 DiloFlow 视频里直接放原片片段（除非 fair use ≤ 30s 各段 + 注明出处 + 已经 rights-ledger 记录）
- 不分享 normalized markdown 给第三方

### 14.3 凭据安全

- Cookie / API Key 只存 `~/.scoutflow/credentials.json`（不进 git，文件权限 600）
- 不把凭据写日志
- ScoutFlow 不做"代登录"代理

### 14.4 频率限制（硬约束）

- 账号源：默认 6 小时一次
- 关键词源：默认 12 小时
- 列表源：默认 24 小时
- 单条链接：触发即查（不轮询）
- 批量采集：≤2 并发，串行 throttle 1s
- LLM 调用：受 LLM 服务商自身限额保护，不做 ScoutFlow 内部硬限

### 14.5 合规 UI

每个 source 配置页必须显示：

- 当前登录方式
- 采集频率
- 是否只读
- 最近一次失败 / 风控记录
- 版权备注框（user 可填）

### 14.6 失败即停

- 连续 3 次 401 / 403 / 429 → source 自动 blocked
- 风控类失败 → 不重试，alert user
- 一旦 blocked，需要 user 手动 enable（避免 silent retry）

### 14.7 数据销毁

- user 删除 capture → 立即删 DB 记录 + FS 目录（不保留）
- user 销毁 source → DB 标 disabled，但 source_items 历史保留
- ScoutFlow 卸载 → 提供脚本一键清理 `data/`
