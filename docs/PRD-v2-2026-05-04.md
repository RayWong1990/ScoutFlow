## 1. Mission / 范围 / 非目标

### 1.1 一句话定义

> ScoutFlow = 把外部内容变成本地可追溯证据、可引用文案资产、可被后续项目消费的 single-user 工作台。

### 1.2 当前目标

- 建立以 SQLite + FS + state words 为核心的 authority-first 基线。
- 把单条 `manual_url` 的 metadata-only 采集闭环做成可信起点。
- 保持 receipt ledger、trust trace、redaction、single-writer 协作纪律可验证。
- 为后续更大 scope 保留清晰 gate，而不是抢跑 runtime。

### 1.3 明确非目标

- 不做 SaaS、多用户、协作权限、分享、邀请。
- 不做浏览器自动化、平台互动、评论/点赞/发布。
- 不自动推进 Phase 2-4 runtime。
- 不把研究 note、draft spec、local probe report 写成 final authority。

### 1.4 当前红线

- `recommendation / keyword / RAW gap` 不得直接创建 capture。
- `audio_transcript` runtime 仍 blocked。
- BBDown runtime、media download、ffmpeg、ASR、frontend、workers 都不因文档 promote 自动解锁。

## 2. 产品边界

### 2.1 单 user / local-first 假设

- 一套本地实例服务一个 user、一组凭据。
- 凭据、QR 图片、auth sidecar、browser profile 路径都不是 evidence。
- 项目默认以本地文件系统和本地 SQLite 为事实源，不追求公网服务或团队 SaaS 流程。

### 2.2 Phase 1A 当前有效范围

- `POST /captures/discover` 只承认 `platform=bilibili`、`source_kind=manual_url`、`quick_capture_preset=metadata_only`。
- `POST /jobs/{job_id}/complete` 只接收当前 phase 允许的 receipt / artifact ledger 证据。
- `GET /captures/{capture_id}/trust-trace` 只做 safe trace，不暗示 media/audio 已批准。
- Auth-present metadata evidence 已通过 `T-P1A-011C` 形成可信样本，但它只证明 metadata probe，不证明 media / ASR runtime。

### 2.3 Phase 2+ 仍是 outline

- `signals / hypotheses / capture_plans / topic_cards` 作为产品结构已经稳定，但当前主要是 Phase 2+ outline，不是本轮 runtime 承诺。
- Recommendation、Signal Workbench、Dispatch UI、RAW Linker 扩展都保留为后续 lane，不在当前 base 中自动落地。

## 3. Authority-first 架构

### 3.1 四层架构

```text
L0 Authority  = SQLite + FS artifact layout + state words
L1 Workers    = stateless execution units
L2 Thin API   = contract enforcement / validation / state transitions / receipt ingestion
L3 Console    = projection layer
```

### 3.2 为什么必须是四层

- 多 agent 协作需要物理上的 contract enforcement 边界。
- Console 不得直接写 authority；所有写都走 API。
- Worker 不持有独家状态，不直接写 SQLite。
- 任何一层下线都不应该把事实留在进程内存里。

### 3.3 当前 promote 后的硬协作纪律

- `LP-001` Capture Scope Gate：推荐、关键词、RAW gap 不能直接采集。
- `LP-006` Single Writer / Multi Reviewer：authority conflict domain 单写者。
- `LP-007` GitHub Audit Source：高风险 / material task 以 PR diff / workflow run 为事实源。
- `LP-SEC-001` Credential Material Is Never Evidence。

### 3.4 任务账本纪律

- `docs/current.md`、`docs/task-index.md`、`docs/decision-log.md` 是 coupled authority surface。
- `docs/specs/contracts-index.md` 管理 contract group / alias / baseline。
- 任务状态和当前焦点必须先写账本，再写其他入口文档。

## 4. Capture Scope Gate 与 quick capture

### 4.1 LP-001 有效口径

- 推荐信号、选题想法、RAW 缺口、关键词扫描结果不得默认直接进入采集。
- 这些来源若要进入采集，必须先经过 scope / plan / approval。
- quick capture 只保留给非常窄的手动入口。

### 4.2 当前 quick capture 有效口径

- 来源必须是 user 手动粘贴的单条 URL。
- 当前实际开放 preset 只有 `metadata_only`。
- `audio_transcript` 虽在旧 amendment 叙事里出现过，但在当前 v2 base 中仍保持 blocked。
- 不允许借 quick capture 顺手扩大到作者扩展、关键词扩展、列表扩展、评论、OCR 或图片原图。

### 4.3 Phase 2 entity outline

以下实体作为稳定产品结构保留，但当前主要是 future outline：

- `signals`
- `hypotheses`
- `capture_plans`
- `topic_cards`

它们的存在用于确保后续 Scope Builder / Signal Workbench 有清晰落点，不等于当前 phase 已批准实现。

## 5. 状态、数据与证据

### 5.1 L0 Authority 三件套

| Authority | 负责 | 不负责 |
|---|---|---|
| SQLite | 结构化数据、状态、关系、索引 | 大对象存储 |
| FS layout | 大对象、evidence 包、日志 append-only | 关系和生命周期表达 |
| state words | 跨 worker / 跨 agent 的语义一致性 | 业务排序或 UI 逻辑 |

### 5.2 当前 capture 主链路

当前仍以 capture 生命周期为工程主链路：

```text
discovered
  -> metadata_fetched
  -> queued
  -> media_downloading
  -> media_ready
  -> audio_extracting
  -> audio_ready
  -> transcribing
  -> transcript_ready
  -> normalizing
  -> doc_ready
  -> indexing
  -> indexed
  -> linking
  -> linked
  -> archived / superseded
```

但 Phase 1A 当前有效闭环只落到 `metadata_fetched`、receipt、trust trace；其余 runtime 状态仍是 future chain，不自动解锁。

### 5.3 receipt / artifact ledger / trust trace

- `artifact_assets` 是当前统一命名，不再使用 `media_assets` 作为 authority 命名。
- receipt 只接收当前 phase 允许的证据种类，不能绕过 API 直写 DB。
- trust trace 必须分层显示：`probe evidence != receipt ledger != capture state != media/audio readiness`。

### 5.4 ToolPreflightResult 与 PlatformResult 分层

- `ToolPreflightResult` 处理本地工具链是否可用。
- `PlatformResult` 只在真实触达平台边界后产生。
- `executable_not_found` 永远不是 `PlatformResult`。
- `T-P1A-009` 的真实结论是本地工具未找到，不是平台访问失败。

## 6. Runtime gates

### 6.1 Metadata probe gate

当前已经证明的成功路径是：

```text
tool_preflight
  -> auth_present_metadata_probe
  -> redacted_parser_classified_evidence
  -> metadata_probe_receipt_bridge
  -> explore_trust_trace
```

有效规则：

- tool preflight 先于任何 `-info` probe
- `-info` probe 先于 receipt
- receipt discipline 先于 trust trace
- `T-P1A-011C` auth-present metadata evidence 是当前成功证据源
- no-auth injected-runner adapter shell 不是 live runtime proof
- no-auth live platform success beyond bounded / injected probe 仍未证明
- 这一窄门不批准 media download、ffmpeg、ASR、manual auth、QR login 或 browser profile 读取

### 6.2 manual auth gate

- QR / manual auth 是独立高风险 gate，不属于 no-auth metadata probe。
- agent 不能替 user 扫码、读取登录态或保存凭据。
- 如果工具会把 auth sidecar 或 QR 图片写到 repo 附近，必须在 repo 外 local-only 路径执行。

### 6.3 audio transcript blocker

`audio_transcript` 仍 blocked。它不能因为：

- BBDown research note
- draft spec
- fixture parser
- local runtime spike report
- auth-present metadata probe

就进入 active runtime。

### 6.4 当前已生效与仍 gated 的区分

- 已生效：tool preflight / metadata probe evidence / receipt baseline / trust trace baseline / redaction baseline
- 仍 gated：真实 BBDown runtime、media、ffmpeg、ASR、workers、frontend、Phase 2-4 runtime

## 7. Worker / API / Console 边界

### 7.1 Worker 边界

- Worker 是 stateless execution unit。
- 读写 authority 只能通过 API。
- Worker 可写约定 FS 路径，但不能自创未约定 path。
- Worker 失败或退出不能把唯一状态留在内存。

### 7.2 Thin API 边界

API 负责：

- validation
- idempotency
- state transition enforcement
- receipt ingestion
- trust trace safe surface

API 不负责：

- 推荐排序策略
- 规整 prompt 内容
- RAW patch 生成策略
- 长任务执行本身

### 7.3 Console 边界

- Console 是 projection layer，不是 authority。
- UI 不得自己决定状态迁移。
- Console 离线不应阻断 Worker + API + ledger 的基本协作。

## 8. 安全与合规

### 8.1 凭据安全

- cookie / token / auth sidecar / signed URL / browser profile path 不得进 Git、PR、CI、logs、DB artifacts、fixtures、tracked docs。
- 任何 evidence 都必须先 redaction，再允许进入 durable path。

### 8.2 平台边界

- 尊重平台 ToS、付费边界、地区边界与低频原则。
- XHS 在当前 phase 仍按更严格只读边界处理。
- 任何需要人工登录的场景都必须显式切换到 manual auth gate。

### 8.3 不自动扩 scope

- 不允许把 recommendation / keyword / RAW gap 写成默认 capture 入口。
- 不允许把一条 approved URL 扩展成账号、列表、关键词的批量采集。

## 9. Phase 路线

### 9.1 当前 Phase 切分

- `Phase 1A`：metadata-only / receipt / trust trace / authority baseline
- `Phase 1B`：未来可能的 RAW link / richer evidence lane
- `Phase 2`：Signal Workbench / Capture Plan / Topic Card
- `Phase 3`：Recommendation / Hermes orchestration
- `Phase 4`：Dispatch UI / 正式多 agent 工作台

### 9.2 当前不自动批准

- `Phase 1B` 及以上任何 runtime
- frontend mode
- dispatch template 抽离
- broader BBDown adapter runtime
