## 2. 总体描述

### 2.1 系统架构（继承 PRD §4 / OM §4）

ScoutFlow 采用 **Authority-first 4 层堆栈**，本 SRD 不重述设计，仅锁约束：

| 层 | 职责 | 可写 | 可读 | SRD 编号 |
|---|---|---|---|---|
| L0 Authority | SQLite + FS + state words | (worker 通过 L2 API 间接写) | 全部 | DR-Schema / DR-FS / DR-States |
| L1 Workers | stateless 业务执行 | 自己写区 + 调 L2 API | L0 view（API 提供） | FR-WRK-* |
| L2 Thin API | contract enforcement | L0（仅它能直写） | L0 全部 | IF-API-* |
| L3 Console | UI projection | **零写权** | L2 API | FR-UI-* |

**架构不变量**（违反即 SRD 拒收）：

| ID | 不变量 | 测试 |
|---|---|---|
| INV-A1 | Console 不直接写 SQLite / FS | TC-INV-001 静态分析无 SQLite/`open(.., 'w')` 出现在 `apps/console/` |
| INV-A2 | Worker 不直连 SQLite | TC-INV-002 grep `apps/`/`workers/` 无 `import sqlite3` 等 |
| INV-A3 | Worker 只写自己 zone | TC-INV-003 `tools/check-worker-zones` |
| INV-A4 | API 不持有产品业务逻辑（推荐排序 / LLM prompt 等） | TC-INV-004 静态分析（人工 audit + spec） |
| INV-A5 | L0 即 L0：FS / DB / state words 三件套联合作真源 | TC-INV-005 启动时一致性自检 |

### 2.6 关键时序：UC-P1-Recommendation-Flow（Phase 3+，Phase 1A 退化版见 UC-P2）

```
participants: Hermes / API / User / Console / Worker(plan_estimator) / Worker(bili) / Worker(asr) / Worker(normalize)

Hermes  -> API: POST /sources/{id}/scan
API     -> DB: insert source_items[N]
API     -> Worker(recommender): trigger recall
API     -> Worker(reranker): trigger rerank
DB      <- recommendations: insert with suggested_action='open_scope'

User    -> Console: 打开 Source Radar
Console -> API: GET /recommendations?status=new
API     -> Console: list

User    -> Console: 点击 [打开 Scope] on rec_id=X
Console -> API: POST /signals (origin_kind='recommendation', origin_id=X)
API     -> DB: insert signals row, status='opened'
Console -> 跳转 Signal Workbench

User    -> Console: 写 hypothesis
Console -> API: POST /signals/{sid}/hypotheses
API     -> DB: insert hypotheses

User    -> Console: 打开 browsing session
Console -> API: POST /browsing-sessions
User    -> 浏览原始页（系统浏览器）
Console -> API: POST /browsing-sessions/{bsid}/events × N

User    -> Console: 在 Scope Builder 配 targets + budget
Console -> API: POST /capture-plans
API     -> DB: insert capture_plans, status='draft'

User    -> Console: 点 [Estimate]
Console -> API: POST /capture-plans/{pid}/estimate
API     -> Worker(plan_estimator): trigger
Worker  -> DB: update capture_plans.estimate_json, status='estimated'

User    -> Console: 看 estimate（容量 / 时间 / 成本）
User    -> Console: 点 [Approve]
Console -> API: POST /capture-plans/{pid}/approve
API     -> 校验 estimate 字段未超 budget
API     -> DB: update status='approved'

User    -> Console: 点 [Run]
Console -> API: POST /capture-plans/{pid}/run
API     -> 创建 N captures（每 target 一个）
API     -> 入 jobs queue（每 capture 一组 jobs）
API     -> DB: signal.status='capturing', plan.status='running'

Worker(bili) -> API: GET /jobs?status=queued&job_type=metadata_fetch
Worker(bili) -> 平台 API
Worker(bili) -> FS: bundle/raw-api-response.json
Worker(bili) -> API: POST /jobs/{id}/complete
... (每个 worker 类似流程，按 6.3 lifecycle 推进)

Worker(normalize) -> API: POST /jobs/{id}/complete (capture lifecycle 进 doc_ready)
Worker(index)     -> API: POST /jobs/{id}/complete (capture indexed)
Worker(raw_linker)-> FS: links/raw-suggestions.jsonl
Worker(raw_linker)-> API: POST /captures/{id}/links + /jobs/{id}/complete (capture linked)
API -> DB: 当 plan 内所有 captures 进入终态 → plan.status='completed', signal.status='evidence_locked'

Console -> API: GET /signals/{sid} → user 看 Topic Card promotion 入口
User    -> Console: promote 选 capture 作 evidence
Console -> API: POST /topic-cards + POST /topic-cards/{tid}/evidence
```

## 3. 功能需求（FR）

每个 FR 必须填表，约定见 §0.2。本节按业务域组织。

### 3.1 Sources & Source Items

#### FR-SRC-001 创建 Source

| 字段 | 值 |
|---|---|
| Title | 创建 source（账号 / 关键词 / 链接 / 列表 / RSS） |
| Description | user 通过 Console 或 agent 通过 candidate 创建 source。dedupe by `(platform, source_type, hash(config_canonical_json))`。 |
| Phase | 0（API 路由 stub）→ 1A（manual_url 单条）→ 2（其他 type 全启用） |
| Source | PRD §7.5.2 / A1.1 §3 |
| Inputs | `source_type ∈ {account,keyword,link,list,rss,manual}` / `platform` / `name` / `config_json` / `polling_interval_seconds`（可空） |
| Outputs | `source` 行 + `id`；status='ready'（除 keyword 默认 'candidate' 等待第一次 scan 验证有结果） |
| Errors | 422 schema 不通过；409 dedupe 已存在；403（Phase 1A 仅允许 type='link' + platform='bilibili'） |
| Acceptance | 创建后 `GET /sources/{id}` 返完整数据；`config_json` 与 §7.5.2.1 schema 一致；audit_log 留痕 |
| Tests | TC-SRC-001（happy）/ TC-SRC-002（dedupe）/ TC-SRC-003（Phase 1A 限制） |
| LP/D | D004（Phase 1 平台限制）/ LP-001 |
| Risks | 关键词源默认 status='candidate' 时 user 可能误以为已 ready |

#### FR-SRC-002 扫描 Source 产生 Source Items

| 字段 | 值 |
|---|---|
| Title | 触发 source scan，写入 source_items |
| Description | API 或 Hermes 触发；source-adapter worker 拉取平台数据；发现的每条对象写 `source_items`，dedupe by `(source_id, platform, platform_item_id)`。 |
| Phase | 1A（仅 link 类型，单条入库）→ 2（account/keyword/list） |
| Source | PRD §7.5.4 / OM §10 |
| Inputs | `source_id`；scan 触发方（user/Hermes/OpenClaw） |
| Outputs | source_items 行 N 条；source.last_checked_at 更新；source.last_success_at（如 OK） |
| Errors | 401/403 认证失败 → source.status='blocked' / `last_error`；429 限流 → 自动 polling × 2 / 连续 3 次 → blocked |
| Acceptance | 扫描完成后 `last_checked_at` 更新；新 source_items.status='new'；不重复入库（dedupe） |
| Tests | TC-SRC-010（fixture B 站 BV）/ TC-SRC-011（dedupe）/ TC-SRC-012（连续 3 次 401 → blocked） |
| LP/D | LP-001（扫描结果不直接 capture） |
| Risks | 限流处理：默认 polling × 2 可能不够保守，Phase 2 实测后调 |

#### FR-SRC-003 source_items 元数据存储

| 字段 | 值 |
|---|---|
| Title | source_items 的 metadata 存储遵循双层（DB + FS） |
| Description | 完整 raw API response 进 FS（如已转化为 capture）；DB 只存索引摘要 + path + sha256（A1.1 §4.3） |
| Phase | 1A |
| Source | A1.1 §4.3 / D027 |
| Inputs | source-adapter worker 产出的 metadata blob |
| Outputs | DB: `raw_metadata_path` / `raw_metadata_sha256` / `metadata_excerpt_json` ≤4KB；FS: `bundle/raw-api-response.json`（仅当 capture 已存在） |
| Errors | excerpt > 4KB → 422；sha256 不匹配 → 503 |
| Acceptance | DB `metadata_excerpt_json` 体积约束 < 4KB；FS sha256 与 DB 一致 |
| Tests | TC-SRC-020（excerpt 大小）/ TC-SRC-021（sha 一致性） |
| LP/D | D027 / 大对象 vs 小对象边界（PRD §5.6） |
| Risks | DB 与 FS 不一致检测靠定期巡检（FR-OPS-005） |

#### FR-SRC-004 Source 状态切换

| 字段 | 值 |
|---|---|
| Title | source 状态机：ready / candidate / blocked / manual / disabled |
| Description | 状态转移由 API 和扫描结果触发；user 可主动 disable / enable |
| Phase | 1A（ready / blocked / disabled）→ 2（candidate / manual） |
| Source | PRD §6.6 / A1.1 §3 |
| Inputs | scan 失败计数 / user PATCH |
| Outputs | source.status / last_error 更新 |
| Errors | 非法状态转移 → 409 |
| Acceptance | source.status 与 scan 失败次数一致；blocked 后不再被 Hermes 触发 |
| Tests | TC-SRC-030 |
| LP/D | — |
| Risks | manual 状态下 user 必须手动操作，UI 必须强提示 |

#### FR-SRC-005 至 FR-SRC-008（删除 / 修改 / 列表 / disable）

| FR ID | Title | Inputs | Outputs | Phase |

### 9.7 故障处理 SOP

#### 9.7.1 worker 崩溃

```text
1. 检查 logs/jobs/{job_type}-{ts}.log 的 stderr
2. 看 jobs 表 status：
   - running → 自动 reset to queued（API 重启时巡检）
   - failed → user 选择 retry 或放弃
3. 如果连续失败 → 检查 source.last_error
4. 若是 LLM 配额 → 等下次或换模型
5. 若是平台风控 → 等冷却 + source.status='blocked'
```

#### 9.7.2 FS 与 DB 不一致

```text
1. tools/check-fs-vs-db 输出 inconsistency 列表
2. 三种情况：
   a. DB 有记录但 FS 缺文件 → reset capture 到 failed_X
   b. FS 有目录但 DB 缺记录 → 写入 data/orphans-{date}.jsonl，user 决定
   c. sha256 不匹配 → 标 capture inconsistent，user 选择重跑
```

#### 9.7.3 DB 损坏

```text
1. SQLite WAL 自动恢复（DR-INT 自带）
2. 若 main DB 损坏 → 从 data/backups/db/ 最近备份恢复
3. 恢复后跑 tools/check-fs-vs-db 检查 FS 一致性
```
