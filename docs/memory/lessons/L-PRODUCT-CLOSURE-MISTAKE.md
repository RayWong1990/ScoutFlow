# Product closure ≠ engineering closure

ScoutFlow Wave 4 工程基线稳 (PR #131-176 全 merge / CI 绿 / 13 surface 静态壳 + Bridge preview) **不等于**产品 closure. 真产品 closure = topic-card-lite + vault preview + RAW handoff 三件套形成的最小可用循环 (用户每天打开 + 信号成长 + raw 入库).

**Why:** post176 复盘核心结论: "PR 很多 / CI 很绿 / 用户不用". `T-PRODUCT-PROOF-NOT-BREADTH` theme + `P-PROOF-PAIR-CANARY` pattern 是 master spec §13 主菜路线 (W2C 真数据接线 + W6J vault commit 解禁) 的纲领.

**How to apply:** 任何 wave 启动前问 "这个 wave 推进的是工程 closure 还是产品 closure?". 工程 closure 不能消耗 prosumer 主菜带宽 (W2C 数据接线 / topic-card-lite proof). 见到 "再加一个 surface / 再扩一个 entity / 再写一份 spec" 但跟 proof loop 无关的提议, 反问 "是否优先 proof pair canary?". 5 overflow lane 守住, 不让 lane spec breadth 拖延 product proof.
