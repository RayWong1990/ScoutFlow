Start with `09_supporting/SOURCE-AVAILABILITY-LEDGER.md`, then `09_supporting/MASTER-MEMORY-ATLAS.md`, then `09_supporting/GRAPH-ADJACENCY-JSON.json`. After that, read cluster indexes or node files based on the task. For route planning, enter through `T-PRODUCT-PROOF-NOT-BREADTH`, `P-PROOF-PAIR-CANARY`, and `E-TOPIC-CARD`. ## Most important conclusion

The strongest post176 direction is not more breadth by default. It is a product proof contraction: use topic-card-lite / vault preview / RAW handoff as a proof pair/canary, while keeping Signal/Hypothesis/CapturePlan as supporting candidate vocabulary until the proof loop demonstrates value. This keeps user-requested max horsepower and strong visual direction, but avoids opening runtime, migration, frontend, or vault true-write gates by implication.

## Limitation

The prompt asks for a full local memory reconstruction over `~/.claude`, handoff trail, claude-mem SQLite/ChromaDB, jsonl transcript sampling, and cross-project memory. Those paths are not available in this sandbox. This ZIP therefore does **not** claim the full local 150-minute pass. It is still useful as a high-coverage candidate atlas over the uploaded ZIP and GitHub connector readback, and it is honest about unavailable evidence.
